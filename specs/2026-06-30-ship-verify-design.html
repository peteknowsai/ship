Bake agentic verification into /ship
prove the feature works before the review reaches Pete

Today ship's REVIEW stage shows Pete the running app and a static read of the diff — nothing actually drives the feature he just built and confirms it works. We add a verifier: an independent sub-agent that exercises the new feature on the running app, screenshots that it works, loops to fix what's broken, and lays the proof into the review card — so Pete reviews evidence it works, not faith. Borrowed from the AI-Builder-Club/skills pr skill; everything in this spec lives inside the ship plugin (peteknowsai/ship).

Why

ship's REVIEW runs /code-review + ponytail-review (both read the diff) and boots the worktree's localhost so Pete can walk it. The only thing judging "does this feature actually do what was intended" is Pete's eyes. That's how the skeleton-shimmer contrast bug slipped through until it was caught by hand. The missing layer: an independent agent (it didn't write the code) that drives the running app and returns a verdict with proof — before Pete is asked to approve.

What we're building — three edits, all in peteknowsai/ship

1New skill: verify

A focused, self-contained skill — "prove the just-built feature actually works." REVIEW invokes it, exactly as it already invokes /code-review and ponytail-review. Keeping it its own file keeps the ship skill lean and lets it be run on its own. (Mirrors how AIBC made pr standalone.)

2Edit: ship REVIEW stage

Invoke verify after the static reviews and before the review card reaches Pete. Fold the verdict + screenshots + taste notes into the card. Broken after the fix-loop → escalate to Pete instead of merging.

3Edit: review-card.html

Add a "proof it works" block (screenshots + verdict) and a "verifier flagged / suggested" block, so the evidence is laid out in the one HTML Pete already reads.

How the verify skill works

Who runs it
A fresh, read-only verifier sub-agent — it did not write the code, so it judges honestly, and app-driving is verbose so context-isolation pays off.
What it does
Drives the running app through the feature's acceptance criteria (pulled from the spec/plan file), screenshots the success state, and returns works / broken / unverifiable plus a short "this looked off" taste note.
Fix loop
broken → the ship driver (Opus) fixes → a fresh verifier re-checks. Cap ~3 rounds. The driver never rubber-stamps its own work — only a fresh verifier can call it works.
Escalation
Still broken after the cap, or unverifiable (e.g. an authed surface it can't reach) → escalate to Pete with the verdict and evidence. It never silently merges a feature it couldn't prove.
Output
Returns { verdict, evidence paths, taste notes } to REVIEW, which folds them into the review card.
REVIEW stage ├─ /code-review + ponytail-review (static — read the diff) ├─ codified gate: tsc · lint · unit · e2e suite (regression — did old stuff break?) ├─ verify ─────────────────────────────────────────────┐ │ fresh verifier drives the feature live │ loop ≤ 3 │ works ✓ → capture screenshots + verdict │ │ broken ✗ → driver fixes → fresh verifier ──────────┘ │ still broken (or unverifiable) → escalate to Pete ├─ render review card (now carries the proof + any flags) └─ GATE: Pete reviews proof → merge / "fix those" / backlog

How the verifier drives the app — Playwright

The driver is Playwright — already a homezero dependency, no new infra. The trick is it's used two ways, which is what resolves "flexible exploration" vs. "deterministic gate":

① Live drive — prove the new feature

The verifier sub-agent drives a real browser through Playwright and clicks through the actual flow like a user — navigate the route, perform the feature's action, observe the result. Being an agent, it finds its way even without pre-written selectors, and judges observed vs. intended. It screenshots the meaningful beats (start → action → success), not a random dump.

② Crystallize — the compounding net

For a core journey, that same exploration is written out as a committed Playwright .spec.ts — a deterministic, headless script that re-runs in the gate forever after. Same tool, so "explore live" becomes the saved spec. That's why it compounds for free, with no spec authored by hand.

Playwright also captures video + trace, but those stay gitignored as debug backup — the card leads with the glanceable screenshots, no video-hosting dance.

What "organized for PM review" means

The screenshots don't land as a folder of PNGs — they go into the review card as a captioned storyboard: the verdict on top, each shot labeled in plain product language. You scan a visual walkthrough, not a test log. Roughly:

The light compounding (kept deliberately small)

When a feature is a genuine core journey — not every little thing — the verifier leaves behind one committed e2e spec for the path it just drove, folded into the feature's own diff. Pete never hand-writes a spec. The standing suite is just "the critical journeys that have shipped," and it's what the codified gate re-runs next time as the regression net. The rule is core flows only — not a spec per micro-feature — so it stays a thin, self-maintaining net, never a maintenance pile.

Pete's role at the gate (unchanged shape, better inputs)

The verifier reports; Pete directs. It never creates work on its own. At the review card Pete sees proof-it-works + any flags, and says one of: merge, "fix those first", or "backlog it." Same two-gate ship shape — he just no longer has to sit down and test whether the thing even works.

The generic seam — what ship names but does not build

The verify skill is repo-agnostic. It assumes the repo hands it a reachable running app, including past auth. It does not implement any auth bypass, session mint, or stack-up — that is the repo's job. This single sentence is the clean seam where a repo plugs in:

  • ship boots the worktree dev server (it already does this) and invokes verify.
  • the repo provides a local/test way to reach authed surfaces.

homezero will later provide that hook (a guarded localhost login). That work is out of scope here — it's a homezero task, not a ship-skill change. Keeping it out is what stops the ship design and the homezero plumbing from tangling.
